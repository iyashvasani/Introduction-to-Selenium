Introduction to Selenium

Manual Testing?

Manual testing mainly involves the physical execution of test cases against various applications to detect bugs and errors

Feature of Manual Testing

  • One of the primitive methods of testing software
  • Do not require the knowledge of a testing tool
  • Can practically test any application
  • Execution of test cases without using automation tools

Limitation of Manual Testing

  • Extremely time-consuming
  • Requires a tester all the time
  • Manual creation of logs and repositories
  • High risk of error
  • Limited scope for testing
  • No support for performance and batch testing

Selenium suite of tools

Selenium Versions

Working principle of Selenium RC (Selenium 2)

Selenium Test Script:

  • Test script is written in any programming language like JAVA, PHP, Python etc.

Remote Control(RC) Server:

  • Selenium Server receives Selenium commands from your test program, interprets them, and reports the results back to the program

Web Browser:

  • The Web browser is injected with Selenium core which interprets and executes the Selenese commands used in the test script

Web Server:

  • The Web browser can now interact with the web server accordingly

Shortcomings of Selenium RC

  • RC consists of an additional server that acts as a middleman
  • This additional setup complicates the architecture
  • The time taken for the execution of commands is longer

Working principle of WebDriver (Selenium 3 and above)

  • Selenium-WebDriver does not use an additional server
  • It makes direct calls to the browser using each browser's native support for automation
  • The execution time for commands is shorter due to the simple architecture

2. Selenium RC
  • Paul Hammant developed Selenium RC
  • Selenium RC is a server written in Java
  • RC makes provision for writing application tests in various programming languages like Java, C#, Perl, PHP, Python etc
  • The RC server accepts commands from the user program and passes them to the browser as Selenium-Core JavaScript commands
  • The browser works accordingly

Drawbacks of Selenium RC

  • RC consists of an additional server that acts as a middleman
  • This additional setup complicates the architecture
  • The time taken for the execution of commands is longer
  • APIs supported by Selenium RC are redundant and confusing
3. Selenium WebDriver
  • Founded by Simon Stewart in 2006
  • It is a cross-platform testing framework
  • Programming interface to create and run test cases
  • Makes provision to act on web elements
  • It does not require a core engine like RC
  • Supports Java, C#, PHP, Python, Perl, Ruby etc.
  • Supports frameworks like TestNG, JUnit, NUnit

Limitations of WebDriver

  • Can only perform sequential testing hence require Grid for parallel testing
  • Cannot test mobile applications. Requires frameworks like Appium
  • Limited reporting. Third-party tools like TestNG are required
  • Limited image testing

4. Selenium Grid
  • Developed by Patrick Lightbody
  • The main objective of Grid is to minimize test execution time
  • Grid was designed to distribute commands to different machines simultaneously
  • Selenium Grid allows the parallel execution of tests on different browsers and different operating systems
  • Grid is exceptionally flexible and is integrated with other suite components for simultaneous execution

Limitations of Grid

  • Limited scalability cannot scale up or down on demand
  • Static since the nodes are pre-configured, making changes is a challenge
  • Not cloud-native Grid cannot use the distributed storage, automatic fail-over facilities
